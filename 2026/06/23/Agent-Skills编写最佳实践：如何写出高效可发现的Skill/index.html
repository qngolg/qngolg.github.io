Agent Skills 编写最佳实践：如何写出高效可发现的 Skill

在 Claude、Cursor 等 AI Agent 平台上，Skill 是一种将领域知识、工作流程和可执行脚本打包复用的机制。一个设计良好的 Skill，能让 Agent 在正确时机自动加载专业知识，稳定完成复杂任务。

但 Skill 写不好，常见问题也很典型：Agent 找不到该用哪个 Skill、加载了却做不对、或者 SKILL.md 太长把上下文挤爆。

本文基于 Claude 官方 Skill 编写最佳实践，结合实战经验，系统梳理 Skill 编写的核心原则、结构规范与常见陷阱。核心结论先行：SKILL.md 要精简，细节往外拆。

一、Skill 是什么？先理解加载机制

在动手写之前，先理解 Skill 的三级加载机制，这直接决定了你应该把内容放在哪里：

这意味着：

内容位置 何时消耗 Token 编写要求
description 启动时始终加载 精准描述「做什么 + 何时用」
SKILL.md 正文 Skill 被触发后加载 保持精简，建议 < 500 行
附属参考文件 按需加载 可以放详细内容
可执行脚本 执行时只消耗输出 优先用脚本代替生成代码

上下文窗口是公共资源。 Skill 与系统提示、对话历史、其他 Skill 元数据共享同一块上下文。写得越精简，Agent 表现越好。

二、核心原则

2.1 简洁至关重要

默认假设：Agent 已经足够聪明。

只添加 Agent 尚不具备的上下文。写每一段内容前，问自己三个问题：

  • Agent 真的需要这个解释吗？
  • 我能否假设 Agent 已经知道这一点？
  • 这段内容的 Token 成本是否值得？

简洁版假设 Agent 已经知道什么是 PDF、库如何工作。不要重复 Agent 的常识。

2.2 设置适当的自由度

根据任务的脆弱性和可变性，匹配指令的具体程度：

自由度 适用场景 写法
高 多种方法都有效，依赖上下文判断 文本指令、启发式步骤
中 有首选模式，允许一定变化 伪代码、带参数的模板
低 操作脆弱易错，一致性至关重要 固定脚本，明确禁止修改

类比： 把 Agent 想象成探索路径的机器人：

  • 两侧悬崖的窄桥（低自由度）：数据库迁移、支付操作 → 给出精确命令，禁止额外参数
  • 开阔平地（高自由度）：代码审查、文档润色 → 给大方向，信任 Agent 找最佳路径

2.3 用目标模型测试

Skill 是对模型的补充，有效性取决于底层模型。应在计划使用的所有模型上测试：

  • 快速模型（如 Haiku）：Skill 是否提供了足够指导？
  • 均衡模型（如 Sonnet）：指令是否清晰高效？
  • 强推理模型（如 Opus）：是否避免了过度解释？

对 Opus 完美的内容，可能需要为 Haiku 补充更多细节。多模型共用时，力求指令对所有模型都适用。

三、SKILL.md 结构规范

3.1 YAML Frontmatter 必填字段

SKILL.md 顶部必须是 YAML frontmatter，包含 name 和 description 两个字段：

name 字段规则

规则 说明
最多 64 个字符 超出会被截断
仅小写字母、数字、连字符 processing-pdfs ✓，Processing_PDFs ✗
不能包含 XML 标签
不能包含保留词 anthropic、claude 等禁用

推荐命名风格：动名词形式（动词 + -ing），清晰描述 Skill 提供的活动：

description 字段规则

规则 说明
必须非空
最多 1024 个字符
不能包含 XML 标签
必须用第三人称 描述会被注入系统提示，人称不一致会导致发现问题
同时说明「做什么」和「何时用」 这是 Skill 被发现的关键

关键认知： 系统可能有超过 100 个 Skill，description 是 Agent 选择正确 Skill 的唯一依据。description 负责「何时用」，SKILL.md 正文负责「怎么用」。

3.2 SKILL.md 正文：精简是指南，不是百科全书

正文建议控制在 500 行以内。 超过时，用渐进式披露拆分到附属文件。

一个精简的 SKILL.md 应该只包含：

  1. Quick Start：最短路径完成任务
  2. 工作流概览：关键步骤和检查清单
  3. 导航链接：指向详细参考文件的链接（仅一层深度）

SKILL.md 里不应该有的内容：

  • Agent 已知的通用概念解释
  • 完整的 API 文档（放 reference/ 目录）
  • 大量示例（放 examples.md）
  • 可执行脚本源码（放 scripts/ 目录，让 Agent 执行而非阅读）

9.2 与 Agent 协作迭代

最高效的 Skill 开发方式：用一个 Agent 写 Skill，用另一个 Agent 测 Skill。

迭代时重点观察：

  • Agent 是否以意外顺序读取文件？→ 结构调整
  • Agent 是否遗漏重要文件引用？→ 链接需更明确
  • Agent 是否反复读取同一文件？→ 内容应移入 SKILL.md
  • Agent 是否从未访问某文件？→ 可能不必要或标示不清

9.3 关注 description 的触发效果

name 和 description 决定 Agent 是否触发 Skill。如果 Skill 该用时没用、不该用时乱用，首先检查 description 是否足够具体。

十、应避免的反模式

反模式 问题 正确做法
SKILL.md 超过 500 行 占用过多上下文 拆分到 reference/ 文件
description 模糊 Agent 无法判断何时触发 写明功能 + 触发场景 + 关键词
用第一/第二人称写 description 系统提示注入时人称混乱 始终用第三人称
嵌套引用（A→B→C） Agent 只预览部分内容 所有引用仅一层深度
解释 Agent 已知的常识 浪费 Token 只写领域特有知识
提供过多工具/库选项 Agent 选择困难 给默认方案 + 例外说明
Windows 风格路径 Unix 环境报错 统一用正斜杠
脚本直接抛异常 Agent 需自行处理错误 脚本内显式处理错误
时效性日期限制 内容很快失效 用「旧模式」折叠区
术语不一致 Agent 理解混乱 全文统一术语
假设依赖已安装 执行失败 明确列出依赖和安装命令

十一、高效 Skill 发布检查清单

发布前逐项核对：

核心质量

  • description 具体，包含关键触发词
  • description 同时说明「做什么」和「何时用」
  • description 使用第三人称
  • SKILL.md 正文在 500 行以内
  • 详细内容拆分到单独文件
  • 无时效性信息（或在「旧模式」区）
  • 全文术语一致
  • 示例具体，非抽象描述
  • 文件引用仅一层深度
  • 工作流步骤清晰，有检查清单

代码与脚本

  • 脚本自行处理错误，不推给 Agent
  • 无魔法数字，常量有说明
  • 依赖包在指令中列出
  • 路径全部使用正斜杠
  • 关键操作有验证步骤
  • 明确脚本是「执行」还是「阅读」

测试

  • 创建了至少 3 个评估场景
  • 在目标模型上测试通过
  • 用真实工作流场景验证
  • Skill 在预期时机正确触发

总结

写好一个 Skill，记住三句话：

  1. description 决定 Agent 找不找得到你 — 第三人称，写清功能和触发场景
  2. SKILL.md 要精简 — 500 行以内，是指南目录不是百科全书
  3. 细节往外拆，脚本往里放 — 参考文件按需加载，可执行脚本比生成代码更可靠

Skill 的价值不在于写得多详细，而在于在正确的时机，给 Agent 恰好够用的信息。
